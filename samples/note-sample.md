## 一个让我重新理解"好工程师"的瞬间

上周有个新同事来找我帮忙看一个莫名的内存泄漏。他的描述是这样的：

> "我跑了一夜，内存涨到 8G 然后崩了。我看了堆栈，每次都是不同的地方。我没头绪。"

我们一起坐下，前 20 分钟全花在**重新陈述问题**上。我问了他几个问题：

1. "8G 是 RSS 还是堆？"——他不确定
2. "崩在不同地方，是 OOM-kill 还是主动 abort？"——也没看
3. "前一天有什么变更？"——没有
4. "这是稳定环境还是预发？数据集多大？"——预发，今天导入了一份新的客户数据

第四个问题问完，他自己脸色就变了——"哦……数据集是平时的 5 倍。"

**问题没换，但描述被翻译了一层后，假设空间从"未知泄漏"缩到了"5 倍数据导致内存峰值"。** 后来定位出来确实是一个 N² 的内存累计，平时数据小看不出来。

## 这个翻译能力其实是核心技能

我之前总觉得"工程师 = 写代码的人"，所以一直在练写代码。但回头看自己处理过的所有难题，**真正卡住我的从来不是某个 API 不会用**，是某段时间里：

- 我没把模糊需求翻译成清晰接口（→ 写出了"什么都能做但什么都做不好"的代码）
- 我没把一堆告警翻译成"系统当前状态"（→ 在事后才看出端倪）
- 我没把 bug 报告翻译成"可证伪的假设"（→ 一直在猜，没在测）
- 我没把代码翻译成"团队下一个人能读懂的故事"（→ 三个月后自己都看不懂）

每一项都是"翻译"。每一项都比"会写 React/Go/Rust"难得多。

## 为什么这事这么难

我猜难是因为它需要**同时持有多种心智模型**：

- 业务那一层：用户到底想干嘛
- 系统那一层：当前架构能/不能承受什么
- 代码那一层：现在这一行字符串实际做了什么
- 团队那一层：谁会维护这段、TA 的背景是什么

普通工程师在写一行代码时只会想"这行能不能 work"。**好工程师在写同一行时会快速跑遍这四层**，然后选一个能让四层都 OK 的写法。这个"快速跑遍"是经验+刻意训练出来的。

## 一个具体的练习：每次 PR 写"为什么这样"，不要只写"做了什么"

这是我自己在用的小习惯。同样一个 PR：

**普通版：**
> 把 `getUser` 改成异步，加了 cache。

**翻译版：**
> 旧 `getUser` 同步阻塞 DB 调用，导致 `/profile` 路由 P95 从 80ms 涨到 320ms（看 metrics）。改成异步 + 5min cache 是因为：(1) 用户名改一次的成本可以接受 5min 延迟；(2) 同样的 cache 模式 `getSettings` 已经在用，团队熟悉。没用 React Query 类客户端方案是因为 SSR 路径上没用，引入会有 bundle size 成本。

后者**让 reviewer 不需要重新跑一遍我的思考**。它把"代码 → 假设 → 决策"翻译成了对方能直接接住的形态。

## 题外话：AI 编程工具加速了什么

我观察到 AI 工具加速最猛的，不是"写代码"——那部分大家吹得太多了。真正让我惊讶的是它**加速了"翻译"环节**：

- 把含糊需求翻译成接口签名（→ Claude 能问出我自己都没想清楚的边界）
- 把一堆日志翻译成可能的根因（→ 它会主动列假设让我排除）
- 把 PR 翻译成 reviewer 能读的故事（→ 让它读 diff 后写解释）

**它没让我的代码能力变强**，但它给了我一个"廉价的翻译伙伴"。这个伙伴会逼我把模糊的脑内活动外化成文字。这个外化过程才是我真正变厉害的原因——不是因为它替我写了代码，是因为它逼我必须把意图说清楚。

## 给自己的总结

下次再有人问我"怎么做才能成为高级工程师"，我可能不会再说"多写代码、多读源码"了。我会说：

> **每次你卡住，先停一下，问自己：现在这个问题，我用同一句话能向四个不同的人都解释清楚吗？**
>
> - 产品经理（业务层）
> - 团队 lead（系统层）
> - 一个刚来的实习生（代码层）
> - 半年后的自己（团队层）
>
> 解释不通的那一层，就是你需要补的能力。

写代码、调 API、读源码——那些是**做翻译的工具**。翻译本身是更稀缺、更核心的事。
